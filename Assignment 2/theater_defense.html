Схема theater
PostgreSQL / DDL + DML

Создал отдельную схему theater — это как папка внутри базы данных, чтобы всё было аккуратно и не смешивалось с другими таблицами. Внутри неё всё: жанры, фильмы, залы, клиенты, сотрудники, сеансы и бронирования.

CREATE SCHEMA IF NOT EXISTS theater;
SET search_path TO theater;
-- теперь не надо каждый раз писать theater.Films, просто Films
🔗
Связи между таблицами
Foreign Keys везде где нужно

Таблицы не изолированы — они связаны через внешние ключи. Например нельзя добавить фильм с жанром которого не существует. База сама проверит и выдаст ошибку. Это защита от мусорных данных.

⚙️
ALTER TABLE после создания
Изменял структуру не пересоздавая таблицы

После создания таблиц добавил новые ограничения, удалил ненужные колонки и добавил новые — всё через ALTER TABLE, без удаления таблиц с данными.

ALTER TABLE theater.Customers DROP COLUMN notes;
ALTER TABLE theater.Customers ADD COLUMN email VARCHAR(100);
ALTER TABLE theater.Customers ADD CONSTRAINT uq_customer_email UNIQUE(email);

Все 11 таблиц

Жанры фильмов. Простая справочная таблица.

Ключевые концепции
Что использовал и зачем
PRIMARY KEY
Уникальный ID каждой строки. Не может повторяться и не может быть пустым. В каждой таблице он один.
FOREIGN KEY
Ссылка на другую таблицу. Нельзя добавить данные со значением которого нет в родительской таблице — база выдаст ошибку.
SERIAL
Автоматически увеличивает ID: 1, 2, 3... Не нужно вводить вручную — база сама считает.
NOT NULL
Поле обязательно к заполнению. Если не указать — база не пропустит запись.
CHECK
Проверяет значение перед вставкой. Например CHECK (age >= 18) — только совершеннолетние сотрудники.
DEFAULT
Значение по умолчанию. Например статус сеанса автоматически ставится 'Scheduled' если не указать.
UNIQUE
Запрещает дублировать значения в колонке. Например два клиента не могут иметь одинаковый email.
GENERATED ALWAYS AS
Вычисляемое поле — годовая зарплата считается сама: salary * 12. Вручную изменить нельзя.
TRUNCATE CASCADE
Очищает таблицы вместе со всеми связанными данными в дочерних таблицах. Быстрее чем DELETE.
Подзапрос в INSERT
Вместо того чтобы угадывать ID, находим его по имени: SELECT film_id FROM Films WHERE name = '...'
